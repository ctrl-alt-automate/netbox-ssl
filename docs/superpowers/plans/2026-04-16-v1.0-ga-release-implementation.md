# Security Policy

## Supported Versions

We provide security updates for the following plugin versions:

| Plugin Version | Supported |
|:--------------:|:---------:|
| 1.0.x          | Yes       |
| 0.9.x          | Yes (bug fixes and security only) |
| 0.8.x and earlier | No (upgrade required) |

See [COMPATIBILITY.md](COMPATIBILITY.md) for the full plugin-to-NetBox version matrix.

## Reporting a Vulnerability

**Please do not report security vulnerabilities through public GitHub issues.**

### Preferred: GitHub Private Vulnerability Reporting

Open a private advisory via:
https://github.com/ctrl-alt-automate/netbox-ssl/security/advisories/new

GitHub will notify the maintainers privately.

### Alternative: Direct email

Send details to the maintainer listed in `pyproject.toml` with the subject
`[netbox-ssl SECURITY]`. Expect an initial acknowledgment within 5 business days.

## What to include

- A description of the vulnerability and its impact
- Steps to reproduce (proof-of-concept, minimal repro)
- Your assessment of severity (CVSS if possible)
- Any suggested remediation

## Our commitment

- Acknowledgment within **5 business days**
- Fix timeline proposal within **14 business days**
- Public disclosure after fix ships, with credit (unless you prefer otherwise)
- A **90-day disclosure window** by default — we will coordinate if you need more or less time

## Out of scope

The following are by-design behaviors and **not security issues**:

- The plugin does not store or manage private keys — it stores only public certificate
  metadata. This is a deliberate trust-boundary choice.
- The plugin is passive — it does not deploy, renew, or actively manage certificates.
- Self-inflicted issues from authenticated NetBox admins (e.g., pasting malformed
  PEM to trigger internal errors) are functional bugs, not vulnerabilities.

## Safe harbor

We will not pursue legal action against researchers who:

- Act in good faith and avoid privacy violations, service disruption, and data destruction
- Give us reasonable time to fix before public disclosure
- Do not exploit a vulnerability beyond what is needed to demonstrate it

## Security updates

Security patches are released as patch-level updates (e.g., 1.0.1). Subscribe to
GitHub releases to get notified: https://github.com/ctrl-alt-automate/netbox-ssl/releases
```

# Tutorial 1: Your First Certificate Import

!!! note "Audience and outcome"
    **You are:** a NetBox admin who has just installed NetBox SSL (or is evaluating it).
    **You will:** import your first certificate, assign it to a Service, and see it
    appear on the expiry dashboard. **Time:** ~10 minutes.

## Prerequisites

- NetBox 4.4.0 or later with NetBox SSL 1.0 installed ([installation guide](../operations/installation.md))
- A PEM-encoded certificate on your clipboard (or use the example below)
- Admin or `add_certificate` permission

## Step 1 — Open the import page

Navigate to **Plugins → SSL Certificates → Certificates**. Click the **Import** button.

![Certificate list with Import button highlighted](../images/certificate-list.png)

## Step 2 — Paste your PEM

Paste the certificate (anything from `-----BEGIN CERTIFICATE-----` through `-----END CERTIFICATE-----`).

If you don't have one handy, here is a self-signed example for practice:

```text
-----BEGIN CERTIFICATE-----
MIID...example...content...
-----END CERTIFICATE-----
```

!!! warning "Private keys are rejected"
    If your PEM contains `-----BEGIN PRIVATE KEY-----` (or any private key header),
    the import will be **refused by design**. The plugin is a public-metadata
    inventory. Private key storage belongs in a dedicated secrets manager.

## Step 3 — Review and save

The plugin parses the PEM and shows a preview:

- Common Name (CN) + Subject Alternative Names (SANs)
- Validity period (`valid_from`, `valid_to`)
- Issuer + chain
- Key algorithm and size
- SHA-256 fingerprint

Click **Save**. Your certificate now has an internal ID.

## Step 4 — Assign to a Service

From the certificate detail view, scroll to the **Assignments** tab and click **+ Add**.

Pick an existing NetBox Service (or Device, or VM) and save. The certificate now
shows up under that Service's certificates.

## Step 5 — See it on the expiry dashboard

Navigate to **Plugins → SSL Certificates → Analytics**. Your new certificate
appears in the expiry forecast and the algorithm distribution chart.

## What just happened

- **Smart Paste** parsed your PEM using the `cryptography` library
- A `Certificate` record was created with public metadata only
- A `CertificateAssignment` linked it to your Service (via a `GenericForeignKey`)
- Every change was logged in NetBox's changelog — including a snapshot of the cert state

"""
Locust load tests for NetBox SSL API.

Run against a local NetBox instance:

    NETBOX_URL=http://localhost:8000 \
    NETBOX_TOKEN=nbt_xxx.yyy \
    locust -f tests/load/locustfile.py --host=$NETBOX_URL

Then open http://localhost:8089 for the web UI.
"""

import os
import random
import string

from locust import HttpUser, between, task

NETBOX_TOKEN = os.environ.get("NETBOX_TOKEN")
if not NETBOX_TOKEN:
    raise RuntimeError("NETBOX_TOKEN env var is required for load tests")


def _generate_sample_pem() -> str:
    """Return a precomputed self-signed PEM.

    Real PEM parsing is expensive; for load tests we want to measure API
    throughput rather than ``cryptography`` library CPU time. We use a
    precomputed PEM and vary the serial by substituting one hex byte.
    """
    import tests.load.seed_data as seed  # lazy import to avoid django setup

    base = seed.SAMPLE_PEM
    suffix = "".join(random.choices(string.hexdigits.lower(), k=4))
    return base.replace("CAFE0042", f"CAFE{suffix}")


class CertificateBrowseUser(HttpUser):
    """Read-heavy user: list + filter + detail views."""

    wait_time = between(1, 3)
    weight = 5

    def on_start(self) -> None:
        self.client.headers.update({"Authorization": f"Token {NETBOX_TOKEN}"})

    @task(5)
    def list_certificates(self) -> None:
        self.client.get("/api/plugins/netbox-ssl/certificates/", name="list")

    @task(2)
    def filter_active(self) -> None:
        self.client.get(
            "/api/plugins/netbox-ssl/certificates/?status=active",
            name="list:status=active",
        )

    @task(2)
    def paginate_page_2(self) -> None:
        self.client.get(
            "/api/plugins/netbox-ssl/certificates/?limit=50&offset=50",
            name="list:paginated",
        )


class CertificateImportUser(HttpUser):
    """Write-heavy user: POST import."""

    wait_time = between(5, 15)
    weight = 1

    def on_start(self) -> None:
        self.client.headers.update({"Authorization": f"Token {NETBOX_TOKEN}"})

    @task
    def import_certificate(self) -> None:
        payload = {"pem_content": _generate_sample_pem()}
        self.client.post(
            "/api/plugins/netbox-ssl/certificates/import/",
            json=payload,
            name="import",
        )


class CertificateDiffUser(HttpUser):
    """Low-volume diff API user."""

    wait_time = between(10, 30)
    weight = 1

    def on_start(self) -> None:
        self.client.headers.update({"Authorization": f"Token {NETBOX_TOKEN}"})

    @task
    def diff_snapshots(self) -> None:
        # Small canned payload; real diffs are large but we measure endpoint
        # round-trip not payload-size throughput.
        self.client.post(
            "/api/plugins/netbox-ssl/certificates/diff/",
            json={"before": [], "after": []},
            name="diff",
        )
