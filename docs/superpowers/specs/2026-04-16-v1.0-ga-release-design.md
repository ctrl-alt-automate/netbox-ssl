# NetBox SSL v1.0.0 GA Release — Design Spec

**Date:** 2026-04-16
**Author:** Elvis (via Claude Opus 4.7 brainstorm session)
**Status:** Approved — ready for implementation
**Target release date:** 2026-04-16 (same-day ship)

## 1. Context and goal

NetBox SSL plugin currently sits at `0.9.0` on `dev` branch. The v1.0 milestone (issues #57-#60) has four release branches (`release/v1.0-docs`, `-migrations`, `-community`, `-stability`) already merged into `dev`. What remains:

- The release itself has not been cut: `pyproject.toml` still reports `0.9.0`, `CHANGELOG.md` has no `[1.0.0]` section, and none of the four milestone issues have been closed.
- Several explicit v1.0 requirements are not yet satisfied: external contributor guide, versioned docs site, semver policy document, load tests, and verified coverage threshold.

**Goal:** ship a polished, verifiably-stable v1.0 GA release with a standout documentation site as the "wow-factor". Scope is deliberately bounded: no new product features. Finalize, polish, ship.

### Out of scope

- New product features (Vault integrations, CT monitoring, AI forecasts, dashboard widgets for NetBox dashboard). These are post-v1.0 roadmap.

- Rewrite of existing reference content (`api.md`, `data-models.md`, `usage.md`). These move into `reference/` with navigation updates only.
- Cross-browser (Firefox, Edge) and mobile docs site verification. Best-effort on Chrome/Safari desktop only.
- Live PyPI publish credentials validation — publish workflow triggers on tag, assumed token is valid. Flagged as pre-check item before tag push.

## 6. Verification and test plan

### Coverage >= 80% (#60 requirement)

1. Baseline: `pytest tests/ -m unit --cov=netbox_ssl --cov-report=term-missing --cov-report=html`
2. Identify modules below 80%: likely `utils/sync_engine.py`, `adapters/*`, `scripts/*`, `management/commands/ssl_diagnostics.py`
3. Write tests until every module >= 80%
4. Stop-condition: if a specific module requires >30 min of work to exceed 80%, document the gap in `docs/development/security-review.md` rather than writing "test theater"
5. Enforce: `[tool.coverage.report] fail_under = 80`
6. Enforce in CI: `pytest --cov-fail-under=80`

### Load test verification (#60 requirement)

1. Run Locust locally against `docker-compose up` NetBox
2. Record results in `docs/operations/load-testing.md` with date, commit SHA, hardware specs

4. Exit criteria: all 4 scenarios meet SLOs

## 7. Execution flow and approval gates

### Principle

Maximum autonomy requested by user. Release gated by **PR + CI + Gemini code review**
mechanisms, not per-step user approval. User intervenes only when serious issues arise.

### Phase A — Local work (autonomous)

1. Create `release/v1.0.0` branch locally
2. Write all new content (17 markdown + 2 python + 1 yaml + 2 workflows + 3 repo-root)
3. Restructure existing docs (move to `reference/`, `operations/`, `development/`)
4. Update README, CHANGELOG, pyproject.toml, `netbox_ssl/__init__.py`
5. Run `pytest --cov`, address gaps until >= 80%
6. Run Bandit, Ruff, `mkdocs build --strict`, `pip-audit`
7. Run Locust locally, document results
8. Logical commits (conventional commit style)

Milestone report: "Local work complete, coverage X%, PR pushed: [URL]"

### Phase B — PR #1 (`release/v1.0.0 → dev`, autonomous)

1. Push branch
2. `gh pr create` with detailed description + closing refs for #57-#60
3. Poll CI via `gh pr checks` (lint, unit-tests, integration tests v4.4 + v4.5)
4. Read Gemini review when it posts (automatic)
5. Address **serious** Gemini findings autonomously (commit + push). Classification:
   - **Serious** (fix before merge): correctness bug, security issue, data integrity risk, breaking change not documented, test regression
   - **Minor** (fix if cheap; otherwise note in PR comment): style preferences, naming suggestions, optional refactors
6. On CI green + Gemini reviewed + serious findings resolved: `gh pr merge --admin --squash`

Milestone report: "PR#1 merged to dev, opening PR#2"

### Phase C — PR #2 (`dev → main`, autonomous)

Same flow as Phase B: push, wait, address, admin-merge.

Milestone report: "PR#2 merged to main, tagging v1.0.0"

### Phase D — Tag and release (autonomous, with pre-checks)

1. Verify PyPI token (`publish.yml` config check)
2. `git tag -s v1.0.0 -m "NetBox SSL v1.0.0 — General Availability"`
3. `git push origin v1.0.0` → triggers `publish.yml` → PyPI publish
4. `gh release create v1.0.0 --notes-file <extract from CHANGELOG>`
5. Monitor `publish.yml` and `docs.yml` workflows to completion

Milestone report: "v1.0.0 live on PyPI, docs site deployed, closing issues"

### Phase E — Post-release (autonomous)

1. Verify `pip install netbox-ssl==1.0.0` works in clean venv
2. `gh issue close #57 #58 #59 #60` with closing comments referencing release
3. Update `MEMORY.md` with v1.0.0 release entry
4. Final report: "v1.0.0 is live. PyPI ✅ Docs ✅ Issues closed ✅. Done."

### Autonomy stop conditions

I halt autonomous execution and ask for guidance when:
- CI fails on a test requiring a design decision (not a mechanical fix)
- Gemini flags a security or data-loss vulnerability requiring architectural change
- PyPI token is expired or invalid
- Unexpected state (merge conflicts from parallel work, missing secrets, etc.)

### Rollback paths

| Stage | Rollback cost |
|-------|---------------|
| Phase A (local) | `git branch -D release/v1.0.0` — free |
| Phase B (PR#1) | `gh pr close`, revert merge commit — low |
| Phase C (PR#2) | Same — low |
| Phase D (tag+release) | Within 72h: `pip yank netbox-ssl==1.0.0`, `git tag -d v1.0.0`, `git push --delete origin v1.0.0`, public communication — **high cost** |
